# Document-Grounded Research

> 当用户提供了一个文档（PPT/PDF/Word/截图/链接）又同时要求做外部调研时，
> 必须先读文档，再搜外部，不能跳过文档直接去搜。

## 核心原则

**1. 文档优先，外部补充**
用户给了文档 = 用户认为文档比网络搜索结果更可信/更准确。
必须先消化文档内容，再用文档中的事实（公司名、产品名、数字、时间线）
作为搜索查询的输入，而不是凭空去搜。

**2. 文档内容是"已知"，搜索是"补充"**
- 文档内容写进了报告要标记为"源自提供材料"
- 搜索结果要标"外部来源"
- 两者冲突时以文档为准（除非有明确可靠的第三方证据）

**3. 先框架，后搜索**
搜索之前先问自己：
- 文档里已经告诉我什么？（逐条列出）
- 文档没告诉我什么？（这些才是搜索方向）
- 搜索关键词来自哪里？（必须来自文档中的实体名/数据/说法）

## 工作流

### Phase 1: 读取文档（不可跳过）

```python
# 文本类文件 → 直接读
# PPT → 提取所有文字内容（slide by slide）
# PDF → pymupdf/marker-pdf 提取
# 图片 → OCR
```

提取完成后，先输出一份"文档要点摘要"给用户确认理解正确：
```
📄 文档要点摘要
- 公司/产品名称：...
- 核心业务：...
- 关键数据：...
- 时间线：...
- 未明确的信息：[列出需要搜索补充的部分]
```

### Phase 2: 提炼搜索方向

基于文档要点，列出3-5个需要外部验证/补充的搜索方向：

```
🔍 需要搜索的方向
1. [方向1] — 因为文档提到X但未涉及Y
2. [方向2] — 因为文档数据截至Z日期，需更新
3. [方向3] — 因为文档提到的竞品需要交叉验证
```

### Phase 3: 搜索 + 合成

搜索时每次查询必须包含文档中提取的实体名作为精确匹配词。
搜索结果与文档内容交叉比对，标注一致/差异。

### Phase 4: 输出

输出结构：
```
## 📄 源自用户文档
{基于文档内容的事实}

## 🔍 外部补充信息
{搜索结果 + 来源标注}

## ✅ 交叉验证结论
{文档 vs 外部的对比/补充/修正}
```

---

## 反模式（必须避免）

- ❌ 用户给了文件，你直接开始搜网络 → 用户不得不纠正你
- ❌ 没读完文档就输出结论 → 文档里写了你却不知道
- ❌ 搜索词不包含文档中的具体实体 → 搜到的是泛泛内容而非定制信息
- ❌ 文档和外部信息混在一起不标注来源 → 用户分不清哪个是你找到的，哪个是已知的
- ❌ 只输出搜索摘要，不结合文档做交叉分析 → 失去了"文档优先"的意义

---

## 用户偏好（猪猪微）

- 中文输出
- 先读文档，再搜外部
- 输出要结构化（分"源自文档"和"外部补充"两块）
- 交叉比对文档与外部的信息一致性
