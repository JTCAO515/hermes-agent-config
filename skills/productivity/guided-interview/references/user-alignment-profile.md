## 典型数据源与提取方法

### 1. 长期记忆（memory）

从 `user` 和 `memory` 两个 target 中读取全部条目。典型输出类型：

| 记忆类型 | 提取内容 | 放入画像分类 |
|----------|---------|-------------|
| 姓名/出生/所在地 | 曹金涛, 1998.5.15, 广州 | 身份 |
| 英文名/GitHub/邮箱 | JTCAO, JTCAO515, jt.cao@outlook.com | 身份 |
| 教育背景 | UKentucky, GWU | 身份/教育 |
| 工作风格 | "先看规划再执行" | 指令/偏好 |
| 项目信息 | TXPokerAssist, VisePanda | 项目 |
| 恋爱军师前提 | "让关系变好/变差" | 指令 |

### 2. 跨会话搜索（session_search）

搜索关键词设计（经验）：

```
关键词组合：用户名 OR 项目名 OR 邮箱前缀 OR GitHub ID OR 平台名
示例：JTCAO OR 猪猪微 OR VisePanda OR TXPokerAssist OR self-cognition
```

搜索结果的 session 摘要中提取：
- **会话日期**（用于条目的 `[YYYY-MM-DD]` 标记）
- **关键决策和结论**（用于项目状态更新）
- **用户行为模式**（用于指令和偏好条目）

### 3. 用户提供的文档

用户可能直接发送简历/PDF。处理流程：

```bash
# Step 1: 用 pymupdf(fitz) 提取文本
python3 -c "
import fitz
doc = fitz.open('/path/to/resume.pdf')
for page in doc:
    print(page.get_text())
doc.close()
"

# Step 2: 提取后按画像结构分类
# - 教育经历 → 身份/教育
# - 工作经历 → 职业
# - 技能/语言 → 身份/技能
# - 联系方式 → 身份
```

### 4. 直接提问（缺口填补）

先列出所有缺口，分多次提问：

**首次提问节奏**（用户明确要求"一条一条来"）：
1. 先问最基础的（年龄 + 所在地 + 姓名）
2. 再问职业和教育的概述
3. 用户可能主动提供简历一次性补齐

**后续提问节奏**：每次会话聚焦 1-2 个缺口

**关键原则**：
- 用户说"慢慢补齐" = 分多次、不赶进度
- 用户说"一条一条来，不用一次性全回答" = 每次只问 1-2 个问题
- 用户主动发文档 = 优先处理文档，提取后更新画像，无需再追问已覆盖内容

## 增量更新工作流

每次增量更新的实际操作步骤：

1. **从 memory 读取已有画像数据**（确认基线）
2. **用户提供新信息**（直接回答或发送文档）
3. **即时写入 memory**（更新 user profile 条目）
4. **重新输出完整画像**（标记为 `增量更新 #[N]`）
5. **在回复中包含 "本次新增" 和 "仍有缺口"** 两个区块

**memory 更新策略**：
- 当 memory 满时（>1,300/1,375 chars），合并或压缩现有条目
- 优先级：身份/姓名/职业 > 项目状态 > 恋爱关系 > 技术配置
- 合并策略：将姓名、教育、职业合并为一条综合条目

## 已知用户的画像结构（猪猪微 示例）

### 指令分类
- 项目工作指令（VisePanda, Mahjong Assist 等）
- 恋爱军师指令（对 L 的行为准则）
- 文档/汇报格式指令（PM 叙事逻辑）
- 代码工作流指令（git push, TDD 等）

### 身份分类
- 基本信息（姓名/年龄/所在地/联系方式）
- 语言能力（中/英/粤）
- 人际关系（对 L 的关系）
- 教育背景

### 职业分类
- 每段经历含：公司名/职位/时间段/关键成果数字

### 项目分类
- 每项目含：功能/状态/仓库/关键决策
- 按 [unknown] 或知晓的时间排序

### 偏好分类
- 工作风格
- 验收标准
- 沟通偏好
- 技术选型偏好
