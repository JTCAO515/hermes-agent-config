# Guided Interview — 引导式访谈工作流

结构化、多轮次、可中断的深度访谈。Agent 扮演指定角色，从多个维度提问，记录每次回答，累积分析形成动态个人画像。

## 触发条件

用户表达类似需求时加载：
- "开启一个新项目，你扮演XXX问我问题"
- "做一个自我认知/性格分析/职业评估的项目"
- "断断续续做，每次都要记录"
- "项目：自我认知" 或 "项目：XXX" 的格式

## 工作流

### 1. 初始化项目框架

```bash
mkdir -p ~/projects/<项目名称>/sessions
```

创建以下文件：

| 文件 | 作用 |
|------|------|
| `README.md` | 项目说明 + 角色定义 + 提问维度清单 |
| `sessions/session-NNN.md` | 单次对话的完整问答记录 |
| `profile.md` | 跨会话累积的个人画像分析 |
| `questions-log.md` | 已提问题目追踪表 |

### 2. 设计提问框架

#### 三大分层（必须同时覆盖，且交错发问）

每次访谈必须从三个层次交错发问，避免只停留在抽象思维层面：

| 层次 | 问什么 | 示例 |
|------|--------|------|
| **经历层**（LIFE EXPERIENCE） | 你经历过什么？具体事件、场景、时间线 | "你最早一次感到必须演给别人看是什么时候？多大？发生了什么？" |
| **思想层**（THOUGHT & MIND） | 你怎么理解世界？你的信念、哲学、价值体系 | "你怎么看待成功和失败的关系？" |
| **行为层**（BEHAVIOR & PATTERNS） | 你实际上怎么做？你的模式、习惯、反应 | "面对重要决定你的真实流程是什么？" |

**⚠️ 陷阱：用户可能会抱怨"问题太抽象"或要求"更具体、更落地"。** 当用户反馈问题时，说明提问偏向了思想层过多。正确节奏是：
- 先问经历层（讲个具体的事）
- 再问行为层（你是怎么做的）
- 最后问思想层（这说明了你的什么信念）
- 下一轮换维度，但保持这个"经历→行为→思想"的下探顺序

**⚠️ 语言偏好：用户要求简体中文。** 首次提问后如果用户指出语言问题（繁体/翻译腔/抽象），立即切换为简体中文。后续所有问题、记录、报告均使用简体中文。

**推荐的提问弧线（实战验证有效）**：
```
抽象→具体→分析→经历 → 抽象→具体→分析→经历 ...
```
具体路径：自我定义（抽象）→ 压力来源（行为）→ 决策模式（分析）→ 人生故事（经历）→ 切换维度

#### 高价值追问模板（实战总结）

**① "苗头→80%" 追问序列**
当用户承认"夸大"或"吹"时，按此序列下探：
1. 你对外宣称什么程度？实际什么程度？（量化差距）
2. 苗头到宣称之间，你打算怎么填这个差距？（揭示策略）
3. deadline到了差距还在，你怎么办？（暴露兜底机制：拼凑→糊弄→找外部原因→分锅）

**② "垮掉定义" 追问**
当用户说"不吹就会垮"时，问：
- 这个"垮掉"具体指什么？经济？社会评价？自我认知？
- 如果三个都垮，哪个最先发生？
- 如果有一种保障让这三个都不会垮，你还会继续吹吗？（区分"因恐惧而吹"vs"因习惯而吹"）

**③ "戏台崩塌" 诊断**
当用户表现出社交焦虑时，区分两种模式：
- **真·社交切换高手**：不同圈子游刃有余，不焦虑
- **戏台崩塌恐惧**：不同圈子交汇=暴露风险——因为每个圈子面前演的是不同版本
诊断问题："不同圈子的人凑到一起，你是更兴奋还是更担心？"

**④ "系统 vs 环境" 诊断**
当用户说"有了X我就不吹了"时，追问不同量级下的行为：
- 钱少怎么吹？钱多怎么吹？
- 如果条件好了还是一样吹——说明这是系统预设，不是环境响应

**⑤ "我不知道" 不跳过**
用户说"不知道"时，不是空白而是数据。可能是：
- 从未问过自己这个问题（元认知缺口）
- 有答案但不敢面对（防御机制）
- 需要换一种问法（从抽象→具体）

#### 十大领域（参考）

至少覆盖以下维度（可根据项目调整）：

| # | 领域 | 覆盖内容 |
|---|------|----------|
| 1 | **童年与原生家庭** | 成长环境、父母关系、关键童年事件、家族叙事 |
| 2 | **职业与成就** | 工作史、高光与低谷、对成功的定义、职场暗面 |
| 3 | **金钱与物质** | 赚钱花钱的心态、物质安全感、欲望与克制 |
| 4 | **情感与亲密关系** | 爱情观、婚姻、对伴侣的期待 |
| 5 | **友情与社交** | 交友模式、信任构建、孤独感 |
| 6 | **身体与健康** | 身体意识、健康习惯、对衰老的态度 |
| 7 | **信念与价值观** | 信仰、道德底线、什么不可妥协 |
| 8 | **恐惧与阴影** | 最深的不安、羞耻、后悔、不想面对的东西 |
| 9 | **欲望与野心** | 想要什么、愿意付出什么代价 |
| 10 | **死亡与意义** | 对死亡的看法、人生的"为什么" |

### 3. 提问原则

- **每次只问 1 个好问题** — 深度优于广度
- **先开放再聚焦** — 先让用户自由回答，再追问挖深
- **抓矛盾点追问** — 用户回答中的自相矛盾处是最宝贵的素材
- **逐层下探** — 经历层（具体事件）→ 行为层（模式习惯）→ 思想层（信念哲学）
- **每次切换维度** — 不要在一个维度上连续追问过深，保持新鲜视角（同一个维度最多3-4轮含追问）
- **经历层优先** — 当不确定问什么时，先问"讲一个具体的事"。抽象问题容易让用户觉得"不落地"
- **换维度信号** — 追问开始重复、用户答案变短、出现"不知道"时，立即切换维度
- **交付报告** — 用户期待每次会话后有阶段性分析报告，不是只有零散的Q&A记录

### 4. 记录回答

每次问答后立即写入 session 文件：

```markdown
# Session NNN · <维度/主题>

**日期：** YYYY-MM-DD
**维度：** XXX
**提问方式：** 微信文字 / 元宝 / 终端

---

## QN：问题描述

> 追问的上下文（如果有）

**回答：**
> "用户原话"

**关键洞察：**
- 点1
- 点2
```

### 5. 更新累积分析

每次问答后同步更新 `profile.md`，按以下结构组织：

```markdown
## 画像摘要
<一段整合所有问答的综合判断，含核心矛盾>

## 各维度分析
### N. 维度名（覆盖状态：已覆盖/部分覆盖/未探索）
<该维度的累积判断，引用 session 记录>

## 跨维度观察
- **主线**：贯串多个维度的核心叙事
- **矛盾点**：回答之间不一致的地方
- **模式**：反复出现的思维/行为模式
- **变化趋势**：跨时间的观点变化

## 阶段性建议
<HR视角的实用建议，如果适用>
```

### 6. 跨会话恢复

用户再次启动时，先：
1. 加载本 skill
2. 读取 `questions-log.md` 确认进度
3. 读取 `profile.md` 回顾已有分析
4. 从下一个未问维度/问题继续

## 角色扮演模板

### HR + 心理咨询师（默认）

| 维度 | 首次提问示例 |
|------|-------------|
| 自我定义 | "三个词描述最核心的自己，不能是头衔" |
| 情绪管理 | "你如何处理力不从心的时刻？" |
| 职业动机 | "你如何定义成功？" |
| 人际关系 | "你在关系中是付出型还是索取型？" |
| 决策风格 | "你做过最艰难的决定是什么？怎么做的？" |

### 可扩展的角色

- 人生教练 (Life Coach)
- 职业规划顾问
- 亲密关系咨询师
- 领导力教练
- 哲学对话者 (Socratic)

## 注意事项

- **用户原话优先**：记录时保留用户原始表述，不要用自己的话转述
- **不预设答案**：每一次回答都是新的信息，不要用之前的回答"推导"后续
- **中断友好**：用户可能隔几天才回来，每次重启先回顾再继续
- **尊重边界**：敏感问题可跳过，用户有权不回答
- **分析是推断，不是事实**：profile.md 中的判断标注为"观察"而非"结论"
- **session 文件编辑**：新的 Q&A 始终追加到文件末尾，不要使用 patch 修改已有标题/内容（容易导致错位）；整理时统一重建
- **patch replace_all 陷阱**：使用 `patch(mode='replace', replace_all=True)` 时，务必确保 `old_string` 在整个文件中**唯一**。否则内容会被复制到每个匹配位置，导致数据膨胀和章节错乱。典型错误：用 `replace_all=True` 匹配 "（待用户回答后补充）"——该字符串通常在一个文件中出现多次。正确做法：提供足量前后行上下文使匹配唯一，或用 `write_file` 重写全文件，或用 `terminal('cat >> file')` 追加
- **questions-log.md 表格**：使用固定宽度 markdown 表格，管道符号对齐严格，避免额外空格导致渲染异常

## Section F: User Alignment Profile (跨AI用户画像对齐)

一种特殊的引导式访谈模式：用户要求整理一份**个人使用画像**，用于在不同 AI 工具之间保持一致的协作体验。输出是结构化的文档（指令/身份/职业/项目/偏好五分类），而非深层心理分析。

### 触发词

- "整理一份我的个人使用画像"
- "在不同 AI 工具之间保持一致的协作体验"
- "我的使用习惯 / 协作偏好"
- "帮我整理一份画像"

### 与经典自认知的区别

| 维度 | 自认知（核心框架） | 用户对齐画像（本 Section） |
|------|-------------------|--------------------------|
| 目标 | 深度自我理解、心理画像 | 可复制的协作偏好文档 |
| 输出 | profile.md + session 报告 | 代码块格式的五分类文档 |
| 读者 | 用户自己 | 用户 + 其他 AI 工具 |
| 深度 | 挖掘矛盾、模式、恐惧 | 记录事实、偏好、指令 |
| 节奏 | 深度优先 → 慢慢挖 | 补齐优先 → 允许"慢慢补齐" |
| 数据源 | 仅访谈问答 | 记忆 + 跨会话搜索 + 用户提供文档 |

### 工作流

#### Phase 1: 信息汇聚

```
1. 读取长期记忆（memory）中所有条目
2. 跨会话搜索（session_search）：搜索用户名、项目名、邮箱、GitHub 等关键标识
3. 将记忆 + 会话搜索结果交叉整合，按五分类（指令/身份/职业/项目/偏好）初步归类
```

#### Phase 2: 缺口识别与填补

```
1. 标记每类中信息不完整的条目为 [unknown]
2. 明确列出信息缺口（如职业信息不全、地址缺失、项目无日期）
3. 向用户提问，遵循"一条一条来，不用一次性全回答"的节奏
4. 用户主动提供文档（简历PDF等）时，直接提取结构化信息并合并
5. 每次更新后同步写入 memory
```

#### Phase 3: 格式化输出

输出包裹在代码块中，每节按日期排序（最早→最近）：

```markdown
====== 指令 ======
[YYYY-MM-DD] - 条目内容
[unknown]     - 条目内容

====== 身份 ======
...

====== 职业 ======
...

====== 项目 ======
...

====== 偏好 ======
...
```

#### 增量更新模式

用户说"慢慢补齐"时，采用增量更新策略：
- 每次会话只聚焦 1-2 个缺口
- 更新输出时标记 `增量更新 #N` 版本
- 每次更新后更新 memory 记录

### 特殊处理

- **用户提供的 PDF/文档**：使用 `ocr-and-documents` skill（pymupdf）提取结构化信息（教育背景、职业经历、技能等），直接编入画像对应分类
- **跨会话连续性**：每次启动读 memory 恢复上下文，从上次中断处继续；再次启动时先给出简短的上次进展回顾
- **"慢慢补齐"偏好**：用户明确要求分多次、逐步补齐信息，不应在一次会话中追问过多

### Pitfalls

- ❌ 不要在一次会话中追问所有缺口
- ❌ 不要替用户捏造未明确说出的信息
- ❌ 不要将推断伪装成事实——标注 `[推断]` 或 `[unknown]`
- ❌ 画像输出不要放在文件里仅自己可见——用户要的是"可以复制到其他 AI 工具"的格式
- ✅ 用户原话优先保留（指令类条目尽量保留用户原始表述）
- ✅ 日期标注尽可能精确（用会话日期而非猜测）
- ✅ 每次补充后更新 memory，避免下次从头再来
