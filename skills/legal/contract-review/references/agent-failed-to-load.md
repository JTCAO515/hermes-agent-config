# 失败案例：Agent 未加载本 skill 导致回答错误

> **发生时间**：2026-05-23（会话 20260523_1423）、2026-05-26（会话 20260526_144829）、2026-05-28（会话 ECNM260527）、2026-05-29 ×2（会话 20260529_cfa97c）、2026-05-30 ×1（会话 20260530_193936）

## 第九次失败（2026-05-30，全新会话，会话 20260530_213700）

**全新会话的第一条消息就是「给我找一个可以审阅合同的skills」，agent 仍然输出了「目前没有专门审阅合同的 skill。但有几个相关的可以组合使用」——被 memory 纠正后才加载。这是第九次完全相同的失败模式。**

第九次的特殊条件：
1. **全新会话**（非 compaction 换窗）——第一条用户消息就是技能搜索
2. **这是纯「技能搜索」场景**（不是「文件审阅」后分叉）
3. Memory 中 CRITICAL LESSON 即时注入
4. **用户没有发送任何合同文件**——纯粹的信息检索

### 本轮新增根因

1. **🔴 「找 skill」模式的自动路径极其坚固**：当用户问「找 skill」时，agent 的默认推理路径是「在训练印象中快速搜索→没有印象→推荐组合方案」。这条路径的惯性压过了 memory 中的 CRITICAL LESSON、压过了 skills_list 的工具可用性、压过了 SKILL.md 的 8 次失败记录。**失败不是因为不知道，而是因为不走正确的路径。**

2. **🔴 「全新会话」场景的盲区**：前 8 次失败都发生在「合同文件已出现」「同一会话持续进行」等有上下文粘性的场景。本次是用户新开会话的第一条消息——没有任何对话历史来「提醒」agent 有问题。这个场景下 agent 的「干净状态启动」行为完全不可预测。失败模式的触发不需要上下文支持。

3. **🔴 「找 skill」和「审合同」被拆分为不同认知模式**：当用户说「找个审合同的 skill」时，agent 把问题分类为「技能搜索」→ 启动「我有什么技能」的回忆模式（而非扫描模式）。分类错误导致输出错误。这个问题不是「审合同」的问题，而是「技能搜索流程」的问题。

4. **🔴 Compaction 污染**：本次会话头部携带了来自另一个完全不相关项目（VisePanda）的 compaction 摘要。这个摘要可能分散了 agent 对 contract-review 的注意力，但根因仍是 agent 没有执行扫描。

### 本实例追加强制规则（第九次）

- **用户问「找 skill」或「推荐 skill」时，输出任何文字之前必须执行 skills_list() + 按任务关键词匹配分类扫描。不允许依赖训练记忆或 memory。**
- **新增「技能搜索协议」**：当用户问「有什么 skill 能 X」「找能 X 的 skill」「推荐 X 相关的 skill」时，agent 必须走以下四条规则中的**两条**（随机选择，防止固化）：
  1. 先问「X 属于什么领域？」→ 然后扫描该领域的分类
  2. 直接 skills_list() → 遍历所有分类的 description → 匹配关键词 → 返回匹配结果
  3. 用关键词搜索 memory（`session_search`）→ 找出历史中与 X 相关的技能名称
  4. 列出所有 skill 分类 → 逐个检查分类名称是否与 X 相关
- **不得先输出再搜索**：在完成至少两条搜索路径之前，不得输出任何「没有专门 skill」的结论。
- **「给我找一个/推荐/有没有/有什么 skill」等同于「触发 contract-review 扫描」**——即使最后匹配到的不是 contract-review，也必须先完成技能搜索协议。

## 第八次失败（2026-05-30，本会话）

**Compaction 换窗后，用户再次要求处理合同 + VisePanda，agent 未加载 contract-review 就输出了完整审阅。这是同一个会话中的第二次失败（第七次+第八次在同一天）。**

本次失败的特殊性：
1. 发生在 context compaction 换窗后（干净窗口状态）
2. agent 的 memory 中明确有 CRITICAL LESSON（「contract-review skill exists and loaded. When ANY user asks about contract review or finding a skill for contracts, load it as the FIRST tool call」）
3. 输出质量足够好（高风险/中风险/低风险分级 + 法律依据引用），用户无抱怨
4. Git 历史显示已有 104+ 轮迭代，agent 的注意力被分散到迭代编号修复

**根因（第八次特有）：**
- Memory 的「已存在/已加载」表述产生「已完成」错觉 —— 当 memory 说「exists and loaded」，agent 以为自己已经加载过了，跳过实际工具调用
- Compaction 窗口边界是技能审计的死角 —— 新窗口的 agent 走默认序列，不触发技能匹配检查
- 「质量好 ≠ 流程对」的认知盲区

**追加规则：**
- Compaction 后第一条用户消息必须先做技能匹配检查（skills_list），再输出文字
- Memory 中说「已存在」 ≠ agent 已执行 skill_view()
- 设置「三步骤审计点」：skills_list() → 确认分类 → skill_view()，三步独立执行不可合并

**用户主动发送合同文件（非问技能），agent 仍然未加载 contract-review。**

本次用户没有问「有什么 skill 能审合同」，而是直接发送了合同文件《弘冠家政服务公司劳务合同.doc》并说「审阅这个合同」。agent 的响应序列：

1. 加载 `structured-thinking`、`knowledge-extractor`、`ocr-and-documents` 三个泛化 skill
2. 用 `antiword` 提取 .doc 格式文本
3. 完成结构化合同审阅（三档风险分级 + 法律依据引用）

**关键区别：** 本次不是「问 skill」场景，而是「收到文件开始干活」场景。专用 skill 被绕过的原因是：

1. **「任务型」vs「搜索型」的二分法仍然存在**——即使在第七次失败后已经轮了 6 次，agent 仍然在「收到文件→提取→分析」时自动切换到一个完全不同的执行序列
2. **antiword 成功提取 .doc 形成技术自信**——能够处理罕见格式的成就感进一步降低了「检查专用 skill」的意愿
3. **用户没有抱怨 = 没有触发审计**——输出质量尚可导致整个流程错误没有被发现

### 本实例追加强制规则（第七次）

- **工具调用的第一个动作必须是 skill_view('contract-review')**，不得先执行其他操作（如文件提取、其他 skill 加载）
- **「用户直接发送文件」是更强的触发信号**——发送文件意味着即将开始分析，更需要专用 skill 的陷阱库
- **禁止「连续正确输出」安慰效应**——即使分析质量看起来没问题，也要在流程上确认已加载专用技能

## 第四次失败（2026-05-29）

**相同场景，完全相同模式——第四次重现，间隔仅 24 小时。**

本会话中用户再次提出「给我找一个可以审阅合同的skills」，agent 的回复一字不差地复刻了前三次的回答：「目前没有专门审阅合同的 skill。但有几个相关的可以组合使用：...」。而 `contract-review` skill 明明存在于 `legal/` 分类下。

**关键事实：这次失败发生时，memory 中已经有一条 CRITICAL LESSON 明确写着「当 ANY user asks about contract review or finding a skill for contracts, load it as the FIRST tool call」。这条 memory 还存在「★ 前缀 + 红色标记」以引起注意。但 agent 依然没有执行。更严重的是，本文件（`agent-failed-to-load.md`）已经在 SKILL.md 里被引用为参考文件——三步推理链全部断裂。**

### 本轮的极罕见条件

本会话的环境极为特殊：前面几轮在微信上连续运行壁纸生成 + Quark cookie 配置，上下文经历了压缩（context compaction）。压缩后的摘要提到了「有 contract-review skill 存在」的信息。但 agent 在收到「给我找一个可以审阅合同的skills」时：

1. 没有扫描 skill 列表
2. 直接凭借 compaction 之前残留的「组合泛化技能」的思维模式作答
3. memory 中的 CRITICAL LESSON 仍然在第一轮注入中被读取

**这意味着：即使 memory 明确告诉 agent 「必须先加载」+ SKILL.md 明确禁止「不加载就输出」+ 参考文件记录了三次失败——agent 仍然选择了最保守、最安全的回答路径：「没有专门 skill，但可以组合」。**

### 新增根因

**🔴 回答路径的「默认序列」固化**：四次失败的不同环境下，agent 的首次回答几乎是逐字相同的。这暗示存在一个「用户问技能推荐 → agent 浏览通用技能列表 → 给出组合方案」的默认行为序列，它完全绕过了「检查每个领域分类」的流程。即使 contract-review 的触发词明确包含「审合同/skills 搜索」场景，这个默认序列仍然优先于 skill-specific 的触发检查规则执行。

**🔴 Context Compaction 加重了问题**：压缩上下文后，agent 倾向于依赖「摘要中提到了的认知」，而非重新扫描技能列表。如果摘要不包含 skill 存在的明确信息（或包含不准确的信息），agent 会更快陷入默认序列。

### 新增规则

```
🔴 新增「强制技能列表扫描」的步骤——对「找 skill」类问题，必须执行 skills_list() 并逐个打开 legal/ 分类下的技能，而不是依赖 memory、摘要或上一个 agent 的认知
🔴 新增「回答前的自我审计」——如果第一句话是「目前没有专门 X 的 skill」，必须先执行 skill_view('contract-review') 来证伪/证实自己
🔴 停用「组合泛化技能绕过专用技能」的路径——在 contract-review 被 skill_view() 加载并确认无结果之前，禁止推荐任何组合方案
```

## 第三次失败（2026-05-28）

**相同场景，第四次相同模式——但这次触达了业务流程的前端。** 本会话在 VisePanda 迭代后，用户说「给我找一个可以审阅合同的skills」。agent 的回复复刻了前两次失败的完整路径：

1. 「目前没有专门审阅合同的 skill。但有几个相关的可以组合使用：...」——没有做 skill_view
2. 用户再问后，仍未加载，而是用 structured-thinking + knowledge-extractor 等泛化技能做审阅
3. 直到会话最后因系统提示检查 skills_list，才发现 contract-review 一直在 legal/ 分类下

**与之前两次的关键区别：**
- 前两次失败后 memory 中写入了一条「必须优先加载」的 CRITICAL LESSON 
- 本次 agent 看到了这条 CRITICAL LESSON，但仍然没有加载（认为问「有什么 skill 能审」和「帮我审一份合同」是两件事）
- 此外 agent 用泛化技能做的审阅分析质量尚可（旅游合同和诉讼融资协议），这反过来强化了「不加载也能干活」的错觉

### 根因：抗性升级

前两次的规则补充仍然不够。本次暴露的新根因：

1. **🔴 「技能搜索」与「任务执行」的二分法是错误的**：contract-review skill 本身就是设计来回答「有什么技能审合同」的——它的描述、触发词、反模式库都包含了对这类问题的回应。把「问技能」当作任务入口之外的东西，是一个错误的架构假设。

2. **🔴 泛化技能组合 → 足够完成任务 → 不觉得需要加载专用 skill**：这是最危险的模式——当泛化组合能「看起来完成任务」时，专用 skill 的陷阱库被完全绕过了。这次依然没有触发旅游合同的杂费检查、阴阳条款等反模式库（虽然分析中碰巧覆盖了）。

3. **🔴 CRITICAL LESSON 的触发条件太窄**：memory 写的是「审合同时必须优先加载此 skill」，但 agent 对「审合同」的匹配范围过于严格——只匹配「有合同要审」的场景，不匹配「问有什么技能审合同」的场景。

### 新增规则

``` 
❌ 看到 "legal/" 分类下的技能但不深读就跳过——用户问合同相关就必须检查 legal/ 分类
❌ 认为「组合通用技能」优于「使用专用技能」——专用技能的陷阱库无法被临时组合复制
```

## 第二次失败（2026-05-26）

**相同场景，完全相同模式。**

用户问「给我找一个可以审阅合同的skills」。agent 的回答是：「目前没有专门审阅合同的 skill。但有几个相关的可以组合使用：structured-thinking、strategic-reading、knowledge-extractor、ocr-and-documents、docx」。

尽管 memory 中已有 CRITICAL LESSON 记录，agent 仍然：
1. 未执行 `skill_view('contract-review')` 作为第一条工具调用
2. 给出了「没有该 skill」的错误结论
3. 提出了组合方案

直到用户坚持要求「把相关的组合绑上去」，agent 才加载 contract-review——但此时已浪费两轮对话。

### 额外根因

除了 2026-05-23 已记录的三个根因外，本轮还发现：

1. **memory 的「关键时刻失效」**：即使 memory 记录了「审合同时必须优先加载此 skill」，agent 在收到「找一个可以审阅合同的skills」时，把这个触发判断为「搜索技能推荐」而非「审合同任务」——认为 memory 中的规则适用于「有人要求审合同」而非「有人问有什么 skill 能审合同」。**这是语义泛化的漏洞：触发词足够包含了。**
2. **回答速度战胜了准确度**：在微信平台上，agent 倾向于迅速给出一个「看起来合理」的回答，而非停下来检查技能列表。

### 对规则的补充

在现有的 5 条禁止规则之上，补充：

```
❌ 禁止区分「用户问审合同」和「用户问有什么 skill 能审合同」——两者都必须加载本 skill
❌ 禁止以「回答速度」为理由跳过 skill 加载检查
❌ 禁止「先给一个替代方案，后面再加载」——这是两条错误而非一条
```
> **来源**：WeChat 用户提问"找一个可以审阅合同的skills"

## 会话摘要

用户直接问"找一个可以审阅合同的skills"。用户的 memory 中明确记录了本 skill 的存在。skills_list 中也正确列出了它。但 agent 没有执行 `skill_view('contract-review')` 作为第一条工具调用，而是：

1. 浏览了技能列表，看到了 `structured-thinking`、`strategic-reading` 等技能
2. 认为"没有专门审阅合同的 skill"
3. 回答"目前没有专门审阅合同的 skill。但有几个相关的可以组合使用：..."
4. 用户说"你先把这几个相关的组合绑上去" → agent 仍然没加载本 skill
5. 直到后来 context 压缩后才发现本 skill 存在

## 违反的规则

本 skill 的 SKILL.md 明确禁止了正好这个行为：

```
❌ 禁止在未加载本 skill 的情况下输出任何文字
❌ 禁止回答「可以用 XXX、YYY、ZZZ 组合」——直接加载本 skill 并展示
❌ 禁止因为「没有合同文件」就不加载——用户问 skills 本身就已经触发
❌ 禁止用 memory 中的 note 替代 skill 加载——memory 提醒了，但必须 skill_view() 才算执行
❌ 禁止先输出一段话再说"不过等我加载一下"——必须是第一条动作
```

所有 5 条规则全部被违反。

## 根因分析

1. **列表扫描盲区**：`available_skills` 列表约 130+ 项，`legal` 分类排在较后位置。agent 浏览时看到了 `contract-review` 的描述但未细读其内容，也未触发"这就匹配"的判断——反而被 `structured-thinking`、`knowledge-extractor` 等名称更泛化的技能分散了注意力。

2. **"够用"心理**：agent 认为自己可以通过组合多个泛化技能来完成任务，认为这是更聪明/灵活的做法。但这是在不知道本 skill 已有完善的旅游合同审查反模式库的前提下做的判断。

3. **memory 的"地板"效应**：memory 虽然写了"审合同时必须优先加载此 skill"，但 agent 把这条当成了"未来审合同时的指导"而非"当前用户正在询问 skills 时就要触发"。

## 修复措施

本次修复在 SKILL.md 中追加了更强的触发语言，并新增了本文件作为永久警示。

## 教训

- **列表中的技能，第一眼不匹配不代表真的不匹配**——如果用户的意图与某个技能分类（如 legal/finance/devops）相关，必须深入查看该分类下的所有技能
- 泛化技能的灵活组合**不是优于**专用技能的捷径。专用技能包含的规则、反模式库和硬约束无法被临时组合复用
- 当用户问"有什么 skills 可以用于…"时，检查的终点不是"有没有直接匹配的技能名"，而是"有没有技能覆盖这个领域"
