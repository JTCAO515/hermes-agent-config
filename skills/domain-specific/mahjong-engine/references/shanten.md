# 向听数算法详解

## 标准面子手向听数的数学公式

```
shanten = 8 - 2*G - M - P

G = 完成面子数（刻子+顺子）
M = 部分组数（搭子+对子，不超过"还需要补的面子数"）
P = 1（有将牌）或 0（无将牌）
```

### 公式推导

14 张胡牌 = 4 组 + 1 将。我们把"完成组数"转换为"还缺几步"：

- 每个完成组贡献 2 步（缺它时 = 差 2 张，已有它时 = 省 2 步）
- 每个部分组贡献 1 步
- 将牌贡献 1 步

从 8 开始减（8 = 4 组 × 2 步 + 将牌 0 步 → 最差情况 8- 向听）。

## 递归回溯算法

### 状态空间

```
arr = [cnt_1, cnt_2, ..., cnt_9, sentinel, sentinel]  (每位 0-4 张)
has_pair = bool  (是否已使用将牌)
```

每个位置有最多 5 个分支（刻子/顺子/对子/搭子-两边/单张），最坏分支数 ≈ 5^9 ≈ 2M。实际上由于排列约束，有效分支远小于此。辅以 `@lru_cache` 后，一个花色通常在 50-500 次递归内完成。

### 分支优先级

```
刻子(3) > 顺子(3) > 对子(2) > 两面搭(2) > 边张搭(2) > 单张(1)
```

各分支消耗的牌数影响其收束速度——消耗越多，递归深度越浅。

### 评分键

```python
def _score_key(value):
    m, p, hp = value  # (melds, partials, has_pair)
    return (m, 1 if hp else 0, p)
```

**必须用自定义键**，理由：

Python 原生元组比较 `(4, 1, False) > (4, 0, True)`：
- 第 1 元素相等 (4 == 4)
- 第 2 元素比较：1 > 0 → 返回 True

但 `(4, 1, False)` 意味着将牌用于了搭子，导致 has_pair=False → shanten = 8 - 8 - 1 - 0 = -1 还是 0？

实际上 shanten = 8 - 8 - min(1,0) - 0 = 0，但正确的应为：
shanten = 8 - 8 - 0 - 1 = -1（因为有将牌）。

评分键修正后：`(4, 0, True) > (4, 1, False)` 因为 `(4, 1, True) > (4, 0, False)` → 第 2 个元素 True(1) > False(0)。
