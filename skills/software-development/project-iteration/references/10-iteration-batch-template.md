# 10 轮迭代批次模板

> 当用户说"自动化迭代N轮"时，按此模板规划和执行。

## 计划阶段

### 1. 确定范围

在开始前先回答：
- 当前项目状态是什么？（读 git log / VISION 文档）
- 上一个停在哪一轮迭代？
- 用户最关心的方向是什么？（新功能 / 修复 / UI / 性能）

### 2. 编写计划文档

创建 `docs/ITERATION_PLAN.md`，表格格式：

```markdown
# [项目名] N轮迭代计划 (Iter X-Y)

| Iter | 模块 | 内容 | 复杂度 |
|:----:|------|------|:------:|
| 81 | 🚄 交通数据 | 高铁路线+航班数据录入 | ⭐⭐ |
| 82 | 🆘 急救卡 | 中文卡片生成 | ⭐⭐ |
| ... | ... | ... | ... |

**范围约束：**
- 所有功能内联到主app（避免Vercel import bug）
- 数据优先用seed/静态，不依赖第三方API
- 每个迭代独立可部署、可验证
```

### 3. 排列原则

- 先易后难（前2轮用 ⭐，建立 momentum）
- 同模块迭代不连续排列（避免审美疲劳）
- 最后1轮留性能/优化类（权重低，容错高）

## 执行阶段

### 每轮循环

```
写代码 → Pre-Deployment验证 → git push → 验证部署 → 向用户汇报
```

**Pre-Deployment 验证（必做）：**
```bash
# 1. 编译检查
python -c "import sys; sys.path.insert(0,'api'); from index import app; print('OK')"
# 2. 数据文件语法
python -m py_compile data/knowledge/*.py
# 3. 路由验证
# 对新增/修改的路由 curl 测试
```

**git push 标准化：**
```bash
git add -A && git commit -m "Iter N: 简要描述" && git push origin main
```

**验证部署：**
```bash
sleep 20 && curl -sL -o /dev/null -w "%{http_code}" https://example.com/
```

**汇报模板：**
```markdown
## 📋 Iter N 完成汇报

### ✅ 本轮完成
- [新增/修改的文件]
- [核心变更]

### 📦 已部署上线
- [部署状态]
- [新功能在哪里查看]

### 🔜 下一轮预期 (Iter N+1)
- [具体目标]
```

### 跳过规则

| 情况 | 处理 |
|------|------|
| 需要 API Key / 凭据 | 跳过，记录到暂停报告 |
| 需要用户在浏览器操作 | 跳过，记录到暂停报告 |
| 功能太大占多轮 | 拆分为多轮，在当前批次完成 |
| 依赖其他先决条件 | 重排序，先做可独立完成的部分 |

## 暂停/完成阶段

用户说"暂停"或N轮执行完后：

### 暂停报告模板

```markdown
## 📊 批次报告

### ✅ 本轮完成 (X轮)

| Iter | 模块 | 状态 | 说明 |
|:----:|------|:----:|------|

### ⏸️ 跳过项

| # | 项目 | 原因 | 需你配合 |
|---|------|------|---------|

### 📈 统计

| 指标 | 值 |
|------|-----|
| 计划迭代数 | N |
| 实际完成 | M |
| 跳过项 | K |
| git commit 数 | M |
| 文件变更 | F |
```

### 复盘要点

- 哪些迭代比预期难？（在 VISION 文档中调整复杂度评级）
- 哪些迭代发现的新坑？（更新 project-iteration 的 deployment pitfalls）
- 用户的偏好/风格是否有新发现？（更新技能或 memory）
