# 脑暴规划 → PLAN.md + ADR → 分批实现流水线

> 适用场景：从产品想法到可执行迭代规划，再到分批落地
> 已验证于 VP-Hermes v4.0.1→v4.0.5（2026-06-19, 6 batches in one session）

## 阶段 1：产出想法（Brainstorming）

使用 `brainstorming` skill 的 Mode A（现有产品）或 Mode B（新产品）：

### 三视角并行

| 视角 | 关注点 | 产出格式 |
|------|--------|---------|
| **PM** | 业务价值、增长、商业化 | 每个方向一个假设 + 用户价值 + 商业价值 |
| **Designer** | 体验、交互、视觉 | 每个方向一个体验升级点 + 开发成本评估 |
| **Engineer** | 技术能力、性能、扩展 | 每个方向一个技术亮点 + 依赖关系说明 |

**每视角产出 5 个想法**（共 15 个），每想法 1-2 行描述即可。

### 流程

1. 理解产品现状（读 CONTEXT.md / README.md / AGENTS.md）
2. 用产品机会空间确定 brainstorm 的边界
3. 逐视角提问：「从这个角度看，什么最有价值？」
4. 不评价、不筛选、先全部产出

---

## 阶段 2：压力测试（Grill-with-docs）

用 `grill-with-docs` skill 对初步想法做压力测试。核心问题：

| 检查项 | 要问的问题 |
|--------|-----------|
| 技术可行性 | 「Vercel 冷启动时这个功能还能用吗？」 |
| 数据完整性 | 「所有城市都有这个字段吗？缺了怎么办？」 |
| 范围控制 | 「MVP 做多大？哪些明确不做？」 |
| 依赖顺序 | 「B2 是否等 B1 的 API 完成？」 |
| 用户感知 | 「用户打开页面第一眼看到什么？」 |

**产出**：修正后的路线图（添加调整说明、风险点、依赖关系）

---

## 阶段 3：锁定范围 → 写 PLAN.md

将修正后的路线图写成 `PLAN.md` 放在项目根目录。

### PLAN.md 结构

```markdown
# [项目名] · 迭代路线图 vX.Y

> 基于 brainstorming 产出 + grill-with-docs 压力测试后修正

## 总体策略

```
一句话轴（如：基础设施先行 → 体验升级 → 决策工具 → 差异化壁垒）
```

---

### Batch N — [标题]

**目标**：[一句话]

**范围**：
- [具体做什么]
- [具体做什么]

**不做**：
- [明确不做的内容，防止范围蔓延]

**估计工时**：[估时]
**版本**：[vX.Y.Z]
**依赖**：[如果有]
```

### 关键原则

- 每个 Batch **独立可部署、可验证**
- 每个 Batch 有明确的 **不做的** 列表（scope boundary）
- 依赖关系用箭头图表示：`B1 → B2, B3`（B2 和 B3 依赖 B1）
- 并行无依赖的 Batch 标记为可同时开发

### 开发顺序图

```
B1a ─→ B1b
       │
B2 ────┘ (并行)
       │
B3 ────┘
       │
B4 ────┘ (随时可插队)
```

---

## 阶段 4：写架构决策（ADRs）

为每个 Batch 的关键决策写 ADR 文档 `docs/adr/NNNN-title.md`。

### ADR 结构

```markdown
# NNNN — [决策标题]

**Status:** Accepted
**Date:** YYYY-MM-DD

## Context

[为什么需要这个决策？当前有什么问题？]

## Decision

[我们决定怎么做？用具体的技术方案描述]

## Consequences

- ✅ [正面影响 1]
- ✅ [正面影响 2]
- ⚠️ [负面影响 / 权衡 / 风险 1]
```

### 需要写 ADR 的典型场景

| 场景 | 示例 |
|------|------|
| 数据存储方案 | SQLite vs Supabase vs JSON 文件 |
| 第三方集成 | 地图引擎选择、支付接入 |
| 范围截断 | 确定 MVP 只覆盖哪几个国家/类别 |
| 架构变更 | 单文件拆分、新增模块 |

---

## 阶段 5：分批实现（Project Iteration）

按 PLAN.md 的 Batch 顺序逐个实现。每个 Batch 的作业流程：

### 单 Batch 实现模板

```
1️⃣ 后端 API 端点（如可独立验证）
   └── curl 或 Python 测试验证
2️⃣ 前端渲染（JS 函数）
3️⃣ 前端样式（CSS）
4️⃣ 前端集成（导航入口、聊天检测、工具卡片点击等）
   └── 本地测试（python3 -m http.server 预览）
5️⃣ 版本号更新（找全所有出现位置）
6️⃣ Changelog 追加
7️⃣ git commit + push
8️⃣ curl 验证部署
9️⃣ 微信汇报（send_message target='weixin'）
```

### 验证门禁（每 Batch）

- [ ] API 端点 curl 返回正确状态码
- [ ] 前端本地预览无报错
- [ ] 无新增 pip 依赖（保持 stdlib only）
- [ ] 旧功能未退化（回归检查）
- [ ] 冷启动场景不会白屏或长时间无响应
- [ ] 版本号已更新（所有出现位置）
- [ ] Changelog 已追加
- [ ] git push 完成
- [ ] 微信汇报已发送

## 已用此模式的案例

| 项目 | 版本范围 | Batches | 说明 |
|------|---------|---------|------|
| VP-Hermes | v4.0.1→v4.0.5 | 6 | Auth→持久化→时间线→城市对比→熊猫表情→签证包 |

## 常见陷阱

- ❌ **跳过 grill 直接写 PLAN** → 缺少技术可行性校验，实现时才发现问题
- ❌ **PLAN.md 的 Batch 不做「不做的」边界** → 范围蔓延，一个 Batch 做太多
- ❌ **ADRs 和 PLAN.md 不一致** → 写决策时改动了范围，但没更新 PLAN.md
- ❌ **多个 Batch 攒一起 push** → 用户要求「每次迭代版本号更新+git push」
- ❌ **版本号只改了 html 忘了改 app.js** → 必须 grep 找全所有出现位置
- ❌ **并行开发的两个 Batch 在同一个 commit 里** → 应该各自 commit 独立部署验证
