# PM Plan-Led Development Workflow (Reference)

> 用户猪猪微明确要求的工作方式：**先写完整产品规划文档，用户确认后再逐模块实施，每步汇报确认**。
> 来源: 2026-06-16 VisePanda-Android 项目启动会话

---

## 核心原则

1. **规划先行** — 写 PM 级产品规划文档（愿景·功能·技术选型·屏幕映射·API契约·工作量估算·风险分析），用户确认后再写代码
2. **Step-by-step 门禁** — 每完成一个步骤，向用户汇报做了什么，等用户确认后再进入下一步
3. **资生产品经理视角** — 文档语气应像资深 PM 写给决策者看：不只是"做什么"，还要解释"为什么选这个方案"、"对比过哪几个选项"、"风险在哪"
4. **用户非技术背景** — 技术决策要用对比表格（Option A/B/C + 理由），不要写代码级细节
5. **工作量透明** — 每个模块附带估算时间，让用户知道投入产出比

## PM 规划文档结构

### 1. 产品定位与愿景

| 痛点 | 解决方案 | 差异化 |
|------|----------|--------|
| 用户实际问题 | 产品怎么解决 | 跟竞品有什么不同 |

### 2. 功能全景与阶段规划

Phase 1 — MVP（功能对等 Web 版）
Phase 2 — 增强功能
每项标注：模块·功能·对标·复杂度·依赖

### 3. 技术架构决策

整体架构图（ASCII 层次图），包含：
- 客户端层
- 数据层
- 后端层

### 4. 关键决策记录 (ADR)

每个重要技术选型用表格对比：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|:----:|
| Option A | ... | ... | ❌ |
| Option B | ... | ... | ✅ 推荐 |

### 5. 屏幕映射与用户流

导航结构 + 底部 Tab + 3-4 个关键用户 Flow

### 6. API 契约映射

Web API → 客户端 Repository → 数据模型的对应关系

### 7. 项目结构

目录树 + 每个文件/目录的职责说明

### 8. 关键实现策略

针对最复杂的 1-2 个模块（如 SSE 流式聊天）的具体方案

### 9. UI/UX 设计系统

品牌色板、字体、设计原则（从 Web CSS 提取）

### 10. 开发计划

实现顺序（依赖关系驱动）+ 逐模块工时估算 + 里程碑

### 11. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:----:|:----:|----------|

## 汇报节奏

每步完成后，向用户发包含以下内容的报告：

```
## ✅ Step N 完成：[模块名]

### 交付清单
| 模块 | 状态 | 说明 |
|------|:----:|------|

### 关键文件
[列出新创建的文件，注明行数]

### 下一步计划：Step N+1 — [模块名]
[一句话描述下一步做什么]

你确认可以继续吗？
```

用户回复"可以的"或"继续"后，才进入下一步。
